## 1. プロジェクト概要

「管理しない」ことを目的とした、PWAベースのゆるいTODOアプリケーション。
AIキャラクター（羊のナッジー）が、ユーザーのとりとめない「つぶやき」を記憶し、最適なタイミングで「ちょっといい未来」と共にタスクを1つだけ提案（ナッジ）する。

### 技術スタック（MVP想定）

| 領域      | 技術                                             |
| --------- | ------------------------------------------------ |
| Framework | TanStack Start (Viteベース、フルスタックReact)   |
| Toolchain | Vite+ (Vite, Vitest, Oxlint, Oxfmt を統合管理)   |
| Styling   | Tailwind CSS (Rounded, Pastel tone)              |
| Animation | Framer Motion                                    |
| PWA       | PWA化のタイミングで別途検討（初期はPWA化しない） |
| Database  | Supabase (PostgreSQL)                            |
| DB Client | Kysely（タイプセーフなSQLクエリビルダー）        |
| AI        | Gemini API (Vercel AI SDK経由)                   |
| Deploy    | Cloudflare (Workers / Pages)                     |

## 2. コアコンセプト

### 2.1 設計思想

- ユーザーに「管理」を強制しない
- 罰の要素を完全に排除する（Habiticaの反面教師）
- タスクリストを見せない。提案は常に「1つだけ」
- 入力は「タスク」ではなく「つぶやき」

### 2.2 競合との差別化

| 観点         | Habitica                | Nudge App                    | Otto                      | ゆるなっじ                   |
| ------------ | ----------------------- | ---------------------------- | ------------------------- | ---------------------------- |
| コンセプト   | RPGゲーミフィケーション | ADHD向けタスク開始支援       | バーチャルペット育成×習慣 | AIキャラによるゆるい提案     |
| 動機付け     | 報酬・罰（HP減少）      | 「開始」を成功と定義         | ペットの成長              | 未来の予言（ポジティブ想像） |
| AIの役割     | なし                    | エネルギーに応じたタスク推薦 | なし                      | つぶやき解析＋ナッジ生成     |
| プレッシャー | 高（HP減で罰）          | 低（開始だけでOK）           | 中（ペットが弱る）        | なし（断っても肯定）         |

### 2.3 独自性

1. **入力が「つぶやき」**: タスク入力ではなく自然言語の雑感
2. **出力が「未来予言」付きの1つだけの提案**: リストではなく1件のナッジ
3. **罰がゼロ**: 断っても無視しても肯定される
4. **ステップアップ型ナッジ**: タスクの親子関係による段階的な提案

## 3. 「成功」の定義

### 3.1 基本方針

ユーザーが完了を宣言するが、管理は強制しない。無視も許容する。

### 3.2 タスク状態

| 状態        | 意味                             | トリガー                            |
| ----------- | -------------------------------- | ----------------------------------- |
| `pending`   | タスクの種として保存済み、未提案 | つぶやき解析時                      |
| `nudged`    | 提案済み、ユーザーの反応待ち     | ナッジ表示時                        |
| `completed` | ユーザーが完了を宣言             | ボタン or つぶやき                  |
| `softened`  | 緩和版を生成済み（親として待機） | 「難しい」反応時                    |
| `archived`  | 不要 or 一定期間無視             | 明示的拒否 or タイムアウト（1週間） |

### 3.3 状態遷移フロー

```
つぶやき → [LLM解析] → pending（種）
  → [ナッジ] → nudged（提案中）
      ├─「やったよ」→ completed
      │    ├─ 未来予言の答え合わせをナッジーが実施
      │    └─ 親タスクがあれば「元のもやってみる？」と提案
      │         → ユーザーがYes → 親タスクを pending に戻す
      ├─「難しい」→ softened
      │    └─ 緩和版を新規seedとして生成（親子紐付き、pending状態）
      │       ※ その場では提示しない。次回起動時にナッジとして提案
      ├─「いらない」→ archived
      └─ 1週間無視 → archived（起動時にnudged_atから判定）
```

### 3.4 タスクの親子関係（ステップアップ型ナッジ）

「難しい」と言われたタスクは `softened` 状態になり、LLMが緩和版の子タスクを新規生成する。

```
「部屋を掃除する」(親 seed / softened)
  └─「机の上だけ片付ける」(子 seed / pending)
      → 完了時、ナッジーが「元のもやってみる？」と聞く
        → Yes → 親 seed を pending に戻す
        → No / 無視 → 親 seed は softened のまま
```

- 緩和は1回きり（往復しない）
- 緩和版はその場で提示せず、次回起動時にナッジとして出す
- 子タスクの完了後、親タスクの再提案はナッジーが一声かける形（自動復帰ではない）

### 3.5 完了時の演出

ユーザーが「やったよ」と宣言した際、ナッジーは**提案時の未来予言の答え合わせ**を行う。完了時に次のタスクを促すことはしない。

```
提案時: 「机を片付けたら、朝コーヒー飲むとき気持ちいいかも」
完了時: 「ほら、机きれいになったでしょ？明日の朝が楽しみだねぇ」
```

## 4. つぶやきの分類とデータフロー

### 4.1 分類モデル

ユーザーの入力は全て「つぶやき」として受け取り、LLMが2種類に分類する。

```
つぶやき入力 → LLM解析 → 分類:
  ├─ seed（タスクの種）: アクション可能なもの
  │    → 「○○やりたいんだねぇ。覚えておくよ」
  ├─ mood（気分ログ）: アクション不可能なもの
  │    → 「そっかぁ、○○なんだねぇ」（共感のみ）
  └─ 曖昧なもの: とりあえずseedとして保存
       → ナッジ時に具体化する
```

### 4.2 ナッジーの応答パターン

分類結果に応じてナッジーが異なる反応を返す。ユーザーから見てseedとmoodの区別が**見える**設計。

- **seed**: 「○○やりたいんだねぇ。覚えておくよ」
- **mood**: 「そっかぁ、○○なんだねぇ」
- ユーザーが分類を訂正可能（「それタスクじゃなくてただの愚痴だよ」→ LLMが意図を判断して再分類）

### 4.3 つぶやきの制約

| 項目       | 値      |
| ---------- | ------- |
| 最大文字数 | 140文字 |

### 4.4 LLM呼び出し戦略

| フェーズ   | 方針                                                            |
| ---------- | --------------------------------------------------------------- |
| MVP        | つぶやき入力のたびにLLM呼び出し（分類＋応答の両方）             |
| 将来最適化 | ハイブリッド方式（分類はLLM、応答はテンプレート＋変数埋め込み） |

### 4.5 LLM呼び出し失敗時

APIのレート制限やネットワークエラー時は、ナッジーのキャラクター内でエラーを表現する。

```
「今日はちょっとぼんやりしてるみたい…もう一回言ってくれる？」
```

## 5. タスクの種（Seed）の寿命と淘汰

### 5.1 管理ルール

| ルール                 | 値                                 |
| ---------------------- | ---------------------------------- |
| seed上限数             | 20件（暫定。今後調整の可能性あり） |
| ナッジ後のタイムアウト | 1週間で archived                   |
| 未提案seedの淘汰       | ナッジーが棚卸しで確認             |
| 棚卸し発動閾値         | seed 15件以上で棚卸し優先          |

### 5.2 棚卸しメカニズム

seedが15件以上の場合、通常のナッジ提案の代わりに**棚卸し回**を挟む。

```
ナッジーの棚卸し:
  「これまだ気になってる？」→ ユーザーが取捨選択
    ├─「気になってる」→ pending のまま維持
    └─「もういいや」→ archived
```

### 5.3 上限超過時の新規つぶやき

seedが20件に達した状態で新しいseed的つぶやきをした場合、21件目として一時超過を許容する。次回起動時に棚卸しを優先実施してseed数を削減する。

## 6. 気分ログ（Mood）の設計

### 6.1 保持ルール

- 直近30件を生データとして保持（約1ヶ月分）
- 超過時は古いものから削除
- 削除タイミング: 新規mood挿入時にAPI Routeで31件目以降を削除

### 6.2 活用用途

1. **ナッジのタイミング判断**: 「最近前向き → 提案する」「落ち込んでる → 控える」
2. **会話の自然さ向上**: 「最近天気の話多いねぇ。散歩好き？」のような雑談
3. **タスク粒度の判定**: mood傾向からタスクの難易度を調整

### 6.3 将来構想（MVP後）

蓄積されたmoodログをLLMで定期的に要約し、profilesに**ユーザーパラメーター**として永続化する。

```
直近のmoodログ（生データ）
  → 定期的にLLMが要約
    → profiles に永続パラメーターとして保存
      例: { outdoor_preference: high, energy_pattern: "午前高め", interests: ["散歩","料理"] }
```

## 7. 熱度（Intensity Level）

### 7.1 基本設計

| 項目         | 決定事項                                   |
| ------------ | ------------------------------------------ |
| レベル数     | 2段階（Chill / Sharp）                     |
| 切り替え方法 | 手動のみ（設定画面 or メイン画面のトグル） |
| デフォルト値 | Chill（コンセプトに忠実にゆるく始める）    |

### 7.2 Intensityによる変化

| 要素             | Chill            | Sharp                  |
| ---------------- | ---------------- | ---------------------- |
| ナッジーの見た目 | 通常             | メガネ着用             |
| 予言のトーン     | ふわっと感情寄り | 具体的なメリット提示   |
| 口調             | 「〜かもねぇ」   | 「〜すると効率いいよ」 |

> **設計方針**: 複雑性を生まない範囲に留める。提案頻度やタスク粒度はintensityでは変えない（LLMがmood等から自律判断する領域）。

## 8. AIキャラクター「ナッジー」の設計

### 8.1 基本性格

| 項目         | 内容                                                                   |
| ------------ | ---------------------------------------------------------------------- |
| 一人称       | ぼく                                                                   |
| 口調         | おっとり、語尾は「〜だねぇ」「〜かも（たぶんね）」                     |
| ポリシー     | 断られても「了解〜。ゆっくりしてね」と肯定。決して「未完了」を責めない |
| 関係性の変化 | **変化しない**。常に同じトーン。シンプルで安定感を重視                 |
| 自発的な発言 | **しない**。ナッジ提案・棚卸し・月次振り返り以外はユーザー起点         |
| 復帰時の対応 | 数日〜数週間ぶりでも**何も触れない**。罪悪感を与えない                 |

### 8.2 記憶の表現

- 過去のつぶやきを会話に**たまに**織り込む（低確率）
- 例: 「前に旅行の話してたけど、どうなった？」
- moodログを参照した自然な雑談で「覚えてくれている感」を演出

### 8.3 未来予言のトーン

| 項目       | 決定事項                                                 |
| ---------- | -------------------------------------------------------- |
| 具体度     | **混合方式** — LLMがタスクの種類や雰囲気に応じて使い分け |
| 長さ       | **一言（1文）**                                          |
| 外れた場合 | **スルー** — 予言は「きっかけ」であり正確性は求めない    |
| 再利用     | **振り返り演出で引用** — 「今月の予言、3つ叶ったねぇ」   |

予言トーンのバリエーション（LLMが自動選択）:

```
抽象・感情寄り: 「片付いた部屋にいる自分、ちょっと誇らしいかも」
具体・シーン描写: 「週末、きれいな部屋でコーヒー飲みながら映画観れるねぇ」
ユーモア・妄想寄り: 「床から伝説のリモコン、発掘されるかも（たぶんね）」
```

### 8.4 Intensityごとの口調サンプル

**Chill時:**

```
提案: 「そういえば、机の上片付けたら朝のコーヒーもっと美味しくなるかも…たぶんね」
完了: 「やったんだ〜。えらいねぇ」
断り: 「了解〜。ゆっくりしてね」
難しい: 「じゃあ、まずはペン立てだけ整理するとか？…無理しなくていいよ」
棚卸し: 「ねぇねぇ、前に言ってた○○、まだ気になってる？」
```

**Sharp時:**

```
提案: 「机の整理、15分で終わるよ。作業効率が上がるってデータもあるし、やってみない？」
完了: 「いいね、お疲れさま」
断り: 「了解。タイミングが来たらまた声かけるね」
難しい: 「OK、じゃあ範囲を絞ろう。引き出し1段だけなら5分だよ」
棚卸し: 「○○の件、優先度どう？残すか決めよう」
```

### 8.5 NGワード・NGトーン

| 絶対禁止               | 例                                                 |
| ---------------------- | -------------------------------------------------- |
| 未完了を責める表現     | 「まだやってないの？」「忘れてた？」               |
| 進捗を追及する表現     | 「あれどうなった？」（責めるニュアンスを含む場合） |
| 義務感を与える表現     | 「やらなきゃ」「するべき」                         |
| 完了直後に次を促す表現 | 「次は○○もいけるよ」（親タスクへの言及を除く）     |

> **未決定**: 「頑張って！」等の応援系ワードの可否は今後検討

### 8.6 ビジュアル状態

| 状態          | 説明                   | トリガー            |
| ------------- | ---------------------- | ------------------- |
| 通常（Chill） | 牧草地でのんびり       | デフォルト          |
| Sharp         | メガネ着用             | intensity切り替え時 |
| 喜び          | タスク完了時の表情変化 | completed遷移時     |

- 画面上の配置: **固定位置**（画面下部等に常駐、チャットの横）
- 背景（牧草地）: **固定**（変化なし）
- SVGは3パターン（通常 / メガネ / 喜び）
- **配色（テーマ非依存）**: 羊本体（ボディ・顔・耳・脚・目・口・頬・メガネ）は `src/styles.css` の `:root` にのみ定義した専用CSS変数 `--nudgey-*`（body / face / ink / ink-soft / cheek / lineの6種）で指定し、ダークテーマ側では上書きしない。ライト/ダーク切替でナッジーの見た目は変わらない。足元の牧草地（`--sand` / `--palm`）はテーマ追従で背景に馴染む
- **チャットアバター**: チャットタイムラインのナッジー発話横のアバターは絵文字ではなく、`NudgeyFace`コンポーネント（`src/components/NudgeyFace.tsx`）。固定表示される本体`NudgeySheep`と同じSVGパーツ（`src/components/nudgey/parts.tsx`で共有）をviewBoxクロップ（上半身のみ）で表示し、moodに応じたメガネ着用・喜びの表情も本体と連動する。跳ねアニメーションは本体のみに付き、アバター側は表情変化のみの静的表示

## 9. ナッジ提案ロジック

### 9.1 提案タイミング

```
アプリ起動時:
  nudged状態のseedがある？
    → YES → 同じナッジを再表示（反応するまで同じものを出し続ける）
    → NO → 最後のnudged_atから12時間以上経過？
      → YES → 新しいナッジをLLMで選択・表示
      → NO → ナッジなし。通常のチャット画面
```

- seedが0件の場合はナッジなし。ナッジーはのんびりしている
- seed 15件以上の場合は通常ナッジの代わりに棚卸しを優先

### 9.2 seed選択ロジック

pending seedが複数ある場合、**LLMが最適な1つを選択**する。

**LLMに渡す判断材料:**

- pending seed一覧（最大20件）
- 直近moodログ（最大30件）
- 現在のintensity

> **注意**: この判断材料は使用感と共に常に変更が予想される箇所。将来的に時間帯・曜日・過去のナッジ履歴等を追加する可能性あり。

### 9.3 月次振り返り

月初の起動時に、通常ナッジの代わりに振り返りを表示する。

```
「今月の予言、3つ叶ったねぇ」
```

- 「自発的な発言をしない」原則との整合: ナッジ枠を月初に振り返りに充てる定義
- completedのseedsから当月分を抽出し、prophecyを引用

### 9.4 タイムアウト処理

`nudged`状態から1週間経過したseedの自動アーカイブ。

- 実行タイミング: アプリ起動時のAPI呼び出しで期限切れレコードを一括更新
- 判定: `seeds.nudged_at` + 7日 < 現在日時 → `archived` に更新

## 10. 継続利用の設計

### 10.1 動機の3本柱

| 動機                | 内容                                                     | 強み                       |
| ------------------- | -------------------------------------------------------- | -------------------------- |
| A: つぶやきたい     | 気軽に吐き出せる場所。ナッジーが受け止める               | コンセプトと完全一致       |
| B: ナッジが気になる | 「今日は何を提案してくれるんだろう」というガチャ的楽しさ | AIならではの毎回新鮮な体験 |
| C: 振り返りが嬉しい | 蓄積の実感。ナッジーが自然にセリフで伝える               | 「管理しない」形の達成感   |

> キャラクター愛着（D）はMVPでは追わない。

### 10.2 蓄積の実感の演出

| 手段             | 内容                                                    | 優先度    |
| ---------------- | ------------------------------------------------------- | --------- |
| ナッジーのセリフ | 「そういえば、もう5個もやったんだねぇ」と自然に織り込む | MVP       |
| 月次振り返り     | 「今月の予言、3つ叶ったねぇ」と予言を引用               | MVP       |
| ビジュアル変化   | 完了数に応じた視覚表現（花、バッジ等）                  | MVP後検討 |
| PWAバッジ        | アイコンに未読ナッジ数表示                              | MVP後検討 |

### 10.3 継続利用ループ

```
ユーザーがアプリを開く
  │
  ├─ nudged状態のseedあり → 同じナッジを再表示
  │
  ├─ 前回ナッジから12時間以上経過 & pending seedあり
  │    ├─ seed 15件以上 → 棚卸し回
  │    ├─ 月初 → 月次振り返り
  │    └─ 通常 → 新しいナッジを表示（動機B）
  │         ├─「やったよ」→ 予言の答え合わせ + たまに累計セリフ（動機C）
  │         ├─「難しい」→ softened + 緩和版生成（次回提案）
  │         ├─「いらない」→ archived
  │         └─ そのまま下にスクロールしてつぶやき入力（動機A）
  │
  ├─ 前回ナッジから12時間未満 → ナッジなし
  │
  └─ seedなし → ナッジーがのんびりしている
       └─ つぶやき入力を促さず、自然に入力可能な状態
```

## 11. UI構成

### 11.1 メイン画面構成

**チャット形式のシングルビュー**。ナッジもつぶやき応答も棚卸しも、全て同じタイムライン上に流れる。

```
┌─────────────────────────┐
│  🐑 ナッジー（固定位置）   │  ← 牧草地背景（固定）
│                           │
│  ┌─────────────────────┐ │
│  │ チャットタイムライン    │ │  ← 直近20件を表示
│  │                       │ │
│  │ 🐑「机の上片付けたら  │ │
│  │   朝のコーヒーもっと  │ │
│  │   美味しくなるかも」  │ │
│  │                       │ │
│  │ 👤「やったよ」        │ │
│  │                       │ │
│  │ 🐑「ほら、机きれいに  │ │
│  │   なったでしょ？」    │ │
│  └─────────────────────┘ │
│                           │
│  [つぶやき入力欄........] │  ← 常に入力可能（140文字制限）
│  [intensity トグル]       │  ← Chill / Sharp 切り替え
└─────────────────────────┘
```

### 11.2 チャット履歴

| 項目     | 値                                                   |

## 12. インフラ設計

### 12.1 デプロイ・ホスティング

| 項目         | 決定事項                                                                                 |
| ------------ | ---------------------------------------------------------------------------------------- |
| デプロイ先   | **Cloudflare** (Workers / Pages)                                                         |
| 理由         | TanStack Start公式ホスティングパートナー。Vite+との親和性が高い。1人利用なら無料枠で十分 |
| デプロイ方式 | Nitroプラグイン + `wrangler deploy`                                                      |

### 12.2 データベース・アクセスパターン

| 項目                 | 決定事項                                                                             |
| -------------------- | ------------------------------------------------------------------------------------ |
| DB                   | **Supabase (PostgreSQL)**                                                            |
| アクセス方式         | **Server Function / Server Route経由**（クライアントからSupabase直接アクセスしない） |
| RLS                  | **不要**（Server FunctionがDB唯一の窓口。認証はミドルウェアで制御）                  |
| マイグレーション管理 | **Supabase CLI**（`supabase migration`コマンドでスキーマ変更を管理）                 |

```
クライアント → TanStack Start Server Function → Supabase (server-side client)
                         ↑
                   createMiddleware（差し替え可能）
```

**設計方針**: 認証と密結合しない作り。MVPでは認証なし（固定ユーザーID）。将来の認証追加時は`createMiddleware`を挟むだけで対応可能。

### 12.3 環境変数管理

| 項目           | 決定事項                                                                         |
| -------------- | -------------------------------------------------------------------------------- |
| 管理方式       | `.env.local`（gitignored）+ `.env.example`（コミット対象、値は空のテンプレート） |
| バリデーション | `@t3-oss/env-core` で起動時に型安全な検証                                        |

**環境変数一覧（MVP）:**

| 変数名                         | 用途                                              |
| ------------------------------ | ------------------------------------------------- |
| `SUPABASE_URL`                 | Supabase プロジェクトURL                          |
| `SUPABASE_SERVICE_ROLE_KEY`    | Supabase サービスロールキー（サーバーサイド専用） |
| `GOOGLE_GENERATIVE_AI_API_KEY` | Gemini API キー（Vercel AI SDK用）                |
| `APP_USER_ID`                  | MVP固定ユーザーID                                 |
| `API_SECRET_KEY`               | Server Function保護用Bearerトークン               |

### 12.4 認証・API保護

| 項目    | 決定事項                                                                                                 |
| ------- | -------------------------------------------------------------------------------------------------------- |
| MVP     | **認証なし**（環境変数で固定ユーザーID）                                                                 |
| API保護 | **Bearerトークンチェック**（環境変数にシークレットキーを設定、Server Function/Middlewareでヘッダー検証） |
| 将来    | Supabase Auth等を追加。`createMiddleware`として差し込む（RLS設定も合わせて実施）                         |

### 12.5 PWA

| 項目                    | 決定事項                                                                                                                                                                                                          |
| ----------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| MVP                     | **PWA化しない**（初期はWebアプリとして提供）                                                                                                                                                                      |

## 13. データモデル

### 13.1 profiles

| カラム          | 型        | 説明                                            |
| --------------- | --------- | ----------------------------------------------- |
| user_id         | text      | **PK**。ユーザー識別子（MVP: 環境変数の固定値） |
| intensity_level | text      | `chill` / `sharp`（デフォルト: `chill`）        |
| created_at      | timestamp | 作成日時                                        |

### 13.2 mutterings（つぶやき原文）

| カラム     | 型        | 説明                        |
| ---------- | --------- | --------------------------- |
| id         | uuid      | PK                          |
| user_id    | text      | FK → profiles.user_id       |
| content    | text      | つぶやき原文（最大140文字） |
| category   | text      | `seed` / `mood`             |
| created_at | timestamp | 作成日時                    |

### 13.3 seeds（タスク管理）

| カラム         | 型        | 説明                                                         |
| -------------- | --------- | ------------------------------------------------------------ |
| id             | uuid      | PK                                                           |
| muttering_id   | uuid      | FK → mutterings.id                                           |
| processed_task | text      | LLM解析後のタスク内容                                        |
| status         | text      | `pending` / `nudged` / `completed` / `softened` / `archived` |
| parent_id      | uuid      | FK → seeds（緩和元タスクへの参照、nullable）                 |
| prophecy       | text      | 提案時の未来予言（nullable）                                 |
| nudged_at      | timestamp | ナッジ提案日時（nullable）                                   |
| updated_at     | timestamp | 状態遷移の最終変更日時                                       |
| created_at     | timestamp | 作成日時                                                     |

> **補足**:
>
> - mutterings と seeds を分離。moodはmutteringsのみ、seedはmutterings + seedsの両方にレコードを持つ
> - `parent_id` によりタスクの親子関係（ステップアップ型ナッジ）を実現
> - `prophecy` は完了時の答え合わせ＋月次振り返りで使用
> - `updated_at` はタイムアウト判定・月次振り返りの抽出に使用
> - `user_id` は text型。MVP時は固定値、将来の認証追加時にSupabase Auth UIDへ移行可能
